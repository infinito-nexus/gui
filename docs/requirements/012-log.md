# 012 - Workspace Audit Logging - Database Storage, Retention & Export

## User Story

As an operator, I want workspace-scoped audit logging with retention, filtering, and export so that I can inspect API activity without polluting workspace files or leaking secrets.

## Definitions

- An "audit event" is one structured record for one completed HTTP request. Long-lived responses such as SSE still produce exactly one audit event for the initial request, not one event per streamed chunk.
- A request is "workspace-scoped" when its routed path or resolved state is bound to one concrete workspace. Non-workspace requests MUST either omit `workspace_id` or set it to `null` consistently across the implementation.
- Actor identity uses `anonymous` for unauthenticated requests and the trusted upstream username / user ID when OAuth2 Proxy is enabled.

## 1. Storage Model

### 1.1 Source of Truth
- [x] Audit logs are persisted in a dedicated database table/collection.
- [x] Audit records are append-only during normal request handling; retention cleanup MAY delete whole records, but no API path mutates an existing audit event in place.
- [x] Storage is tenant/workspace-scoped (`workspace_id` required).
- [x] Audit logs are not stored in `inventory.yml`, `host_vars`, or `group_vars`.
- [x] Audit logs are not stored as workspace `logs/` files by default.

### 1.2 Git & Workspace Integration
- [x] Audit records are not committed to workspace Git history.
- [x] Workspace autosave/history commits do not include audit records.
- [x] Workspace ZIP export excludes audit records by default.

### Acceptance Criteria
- [x] API actions are queryable from DB without reading workspace files.
- [x] No audit-log artifacts are required in workspace file tree.
- [x] No `git -> log -> git` recursion can occur.

## 2. What Must Be Logged
- [x] Every backend API request that is loggable under the active workspace policy creates exactly one audit event when the response reaches its terminal HTTP status.
- [x] Each event includes a UTC ISO 8601 timestamp with a `Z` suffix.
- [x] Each event includes `workspace_id` when the request is workspace-scoped; otherwise the field is omitted or `null` consistently.
- [x] Each event includes HTTP method and path.
- [x] Each event includes HTTP status code.
- [x] Each event includes duration in ms.
- [x] Each event includes actor identity:
  - [x] `anonymous`
  - [x] authenticated username/user_id from trusted header
- [x] Each event includes client IP when available; otherwise it stores the canonical fallback value `unknown`.
- [x] Optional request/correlation ID is supported and is either omitted or set to `null` consistently when absent.
- [x] Optional user-agent is supported and is either omitted or set to `null` consistently when absent.

### Sensitive Content Rules
- [x] Plaintext secrets are never written to audit records.
- [x] Vault passwords are never written to audit records.
- [x] Sensitive fields are masked using the same masking rules as SSE and persisted job logs from [014-e2e-dashboard-deploy.md](014-e2e-dashboard-deploy.md) and [017-security-hardening.md](017-security-hardening.md).

### Acceptance Criteria
- [x] One structured audit event exists per API action.
- [x] No plaintext secrets appear in persisted audit data.

## 3. Audit Event Format
- [x] Event payload is structured JSON.
- [x] JSON is UTF-8 encoded.
- [x] No ANSI formatting is stored.

### Example Event
```json
{
  "timestamp": "2026-02-24T18:22:33Z",
  "workspace_id": "abc123",
  "user": "anonymous",
  "method": "POST",
  "path": "/api/workspaces/abc123/files",
  "status": 200,
  "duration_ms": 42,
  "request_id": "3ed0f4ec-8f3d-44a5-b9d4-e2fb58fd3dc7"
}
```

## 4. Retention & Cleanup
- [x] Retention period is configurable per workspace as `retention_days`.
- [x] Default retention is 180 days (6 months).
- [x] Events older than retention are deleted automatically.
- [x] Cleanup runs as a scheduled/background task and does not require manual invocation.
- [x] Cleanup is workspace-safe and deterministic.

### Acceptance Criteria
- [x] Expired events are removed automatically.
- [x] Active requests are not blocked by cleanup.

## 5. Configurable Logging Policy
- [x] Logging configuration is stored per workspace in DB.
- [x] Config updates apply immediately to newly completed requests; in-flight requests MAY complete under the policy snapshot active when they started.
- [x] Policy supports selecting one primary logging mode plus endpoint exclusions.

### 5.1 API Filter Configuration
- [x] Default mode logs all API calls (`all`).
- [x] Filter mode `writes-only` logs only `POST`, `PUT`, `PATCH`, and `DELETE` requests.
- [x] Filter mode `auth-only` logs only authentication/session endpoints.
- [x] Filter mode `deployment-only` logs only deployment creation, cancellation, status, and log-stream bootstrap endpoints.
- [x] Filter mode `errors-only` logs only responses with status `>= 400`.
- [x] Specific endpoints (for example health checks) can be excluded.
- [x] Filtering is deterministic and workspace-scoped.

### Acceptance Criteria
- [x] Config is persisted per workspace.
- [x] Health endpoints can be excluded.

## 6. Backend API
- [x] `GET /api/workspaces/{id}/logs/config`
- [x] `PUT /api/workspaces/{id}/logs/config`
- [x] `GET /api/workspaces/{id}/logs/entries`
- [x] `GET /api/workspaces/{id}/logs/entries/export`

### Rules
- [x] Config updates are validated.
- [x] Entry listing supports pagination via explicit `page` and `page_size` parameters and returns newest-first by default.
- [x] Entry listing supports filters (`from`, `to`, `user`, `ip`, `q` search string, status, method).
- [x] `from` and `to` are inclusive UTC timestamps.
- [x] `q` searches only non-secret textual fields such as path, user, user-agent, and request ID.
- [x] Export supports at least JSONL and CSV.
- [x] Export can return ZIP for large datasets.
- [x] ZIP export contains exactly one data file in the selected format and MAY include an additional metadata manifest.
- [x] Export endpoint supports the same filters (`from`, `to`, `user`, `ip`, `q`, `status`, `method`) as listing.
- [x] No endpoint exposes audit data from other workspaces.

## 7. UI Behavior

### 7.1 Audit Logs View
- [x] Audit logs are shown in a dedicated "Audit Logs" view (not as file tree folder).
- [x] View supports filtering by date range (`from`/`to`), user, IP, search string, status, and method.
- [x] View supports pagination.
- [x] View supports export action.
- [x] Export uses the currently selected filters (`from`, `to`, `user`, `ip`, `q`, `status`, `method`).

### 7.2 Configuration Modal
- [x] Field: retention duration in days.
- [x] Field: logging mode selector (`all`, `writes-only`, `auth-only`, `deployment-only`, `errors-only`).
- [x] Field: exclude-health-endpoints toggle.
- [x] Actions: Save and Cancel.

### Acceptance Criteria
- [x] Changes are saved via API.
- [x] Invalid configurations are rejected with clear errors.

## 8. Security & Privacy
- [x] No plaintext secrets are ever written.
- [x] Masking is identical to SSE masking behavior.
- [x] Audit records are workspace-scoped.
- [x] No cross-workspace visibility is possible.
- [x] Access to audit read/export/config endpoints is RBAC-protected.
  - [x] Anonymous sessions can access only the audit data of the workspace bound to their current session.
  - [x] Authenticated users can access only workspaces they own, unless a future higher-order role is introduced explicitly.

### Acceptance Criteria
- [x] Secret leakage through audit logging is prevented.
- [x] Workspace isolation is enforced for read and export.

## 9. Performance Requirements
- [x] Audit writing is non-blocking with respect to the request path; long-lived responses such as SSE MUST NOT emit per-chunk audit writes.
- [x] DB writes do not materially delay API responses.
- [x] Cleanup/export do not block active requests.
- [x] Background jobs handle heavy cleanup/export work.
