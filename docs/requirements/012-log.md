# 012 - Workspace Audit Logging - Database Storage, Retention & Export

## User Story

As an operator, I want workspace-scoped audit logging with retention, filtering, and export so that I can inspect API activity without polluting workspace files or leaking secrets.

## Definitions

- An "audit event" is one structured record for one completed HTTP request. Long-lived responses such as SSE still produce exactly one audit event for the initial request, not one event per streamed chunk.
- A request is "workspace-scoped" when its routed path or resolved state is bound to one concrete workspace. Non-workspace requests MUST either omit `workspace_id` or set it to `null` consistently across the implementation.
- Actor identity uses `anonymous` for unauthenticated requests and the trusted upstream username / user ID when OAuth2 Proxy is enabled.

## 1. Storage Model

### 1.1 Source of Truth
- [ ] Audit logs are persisted in a dedicated database table/collection.
- [ ] Audit records are append-only during normal request handling; retention cleanup MAY delete whole records, but no API path mutates an existing audit event in place.
- [ ] Storage is tenant/workspace-scoped (`workspace_id` required).
- [ ] Audit logs are not stored in `inventory.yml`, `host_vars`, or `group_vars`.
- [ ] Audit logs are not stored as workspace `logs/` files by default.

### 1.2 Git & Workspace Integration
- [ ] Audit records are not committed to workspace Git history.
- [ ] Workspace autosave/history commits do not include audit records.
- [ ] Workspace ZIP export excludes audit records by default.

### Acceptance Criteria
- [ ] API actions are queryable from DB without reading workspace files.
- [ ] No audit-log artifacts are required in workspace file tree.
- [ ] No `git -> log -> git` recursion can occur.

## 2. What Must Be Logged
- [ ] Every backend API request that is loggable under the active workspace policy creates exactly one audit event when the response reaches its terminal HTTP status.
- [ ] Each event includes a UTC ISO 8601 timestamp with a `Z` suffix.
- [ ] Each event includes `workspace_id` when the request is workspace-scoped; otherwise the field is omitted or `null` consistently.
- [ ] Each event includes HTTP method and path.
- [ ] Each event includes HTTP status code.
- [ ] Each event includes duration in ms.
- [ ] Each event includes actor identity:
  - [ ] `anonymous`
  - [ ] authenticated username/user_id from trusted header
- [ ] Each event includes client IP when available; otherwise it stores the canonical fallback value `unknown`.
- [ ] Optional request/correlation ID is supported and is either omitted or set to `null` consistently when absent.
- [ ] Optional user-agent is supported and is either omitted or set to `null` consistently when absent.

### Sensitive Content Rules
- [ ] Plaintext secrets are never written to audit records.
- [ ] Vault passwords are never written to audit records.
- [ ] Sensitive fields are masked using the same masking rules as SSE and persisted job logs from [014-e2e-dashboard-deploy.md](014-e2e-dashboard-deploy.md) and [017-security-hardening.md](017-security-hardening.md).

### Acceptance Criteria
- [ ] One structured audit event exists per API action.
- [ ] No plaintext secrets appear in persisted audit data.

## 3. Audit Event Format
- [ ] Event payload is structured JSON.
- [ ] JSON is UTF-8 encoded.
- [ ] No ANSI formatting is stored.

### Example Event
```json
{
  "timestamp": "2026-02-24T18:22:33Z",
  "workspace_id": "abc123",
  "user": "anonymous",
  "method": "POST",
  "path": "/api/workspaces/abc123/files",
  "status": 200,
  "duration_ms": 42,
  "request_id": "3ed0f4ec-8f3d-44a5-b9d4-e2fb58fd3dc7"
}
```

## 4. Retention & Cleanup
- [ ] Retention period is configurable per workspace as `retention_days`.
- [ ] Default retention is 180 days (6 months).
- [ ] Events older than retention are deleted automatically.
- [ ] Cleanup runs as a scheduled/background task and does not require manual invocation.
- [ ] Cleanup is workspace-safe and deterministic.

### Acceptance Criteria
- [ ] Expired events are removed automatically.
- [ ] Active requests are not blocked by cleanup.

## 5. Configurable Logging Policy
- [ ] Logging configuration is stored per workspace in DB.
- [ ] Config updates apply immediately to newly completed requests; in-flight requests MAY complete under the policy snapshot active when they started.
- [ ] Policy supports selecting one primary logging mode plus endpoint exclusions.

### 5.1 API Filter Configuration
- [ ] Default mode logs all API calls (`all`).
- [ ] Filter mode `writes-only` logs only `POST`, `PUT`, `PATCH`, and `DELETE` requests.
- [ ] Filter mode `auth-only` logs only authentication/session endpoints.
- [ ] Filter mode `deployment-only` logs only deployment creation, cancellation, status, and log-stream bootstrap endpoints.
- [ ] Filter mode `errors-only` logs only responses with status `>= 400`.
- [ ] Specific endpoints (for example health checks) can be excluded.
- [ ] Filtering is deterministic and workspace-scoped.

### Acceptance Criteria
- [ ] Config is persisted per workspace.
- [ ] Health endpoints can be excluded.

## 6. Backend API
- [ ] `GET /api/workspaces/{id}/logs/config`
- [ ] `PUT /api/workspaces/{id}/logs/config`
- [ ] `GET /api/workspaces/{id}/logs/entries`
- [ ] `GET /api/workspaces/{id}/logs/entries/export`

### Rules
- [ ] Config updates are validated.
- [ ] Entry listing supports pagination via explicit `page` and `page_size` parameters and returns newest-first by default.
- [ ] Entry listing supports filters (`from`, `to`, `user`, `ip`, `q` search string, status, method).
- [ ] `from` and `to` are inclusive UTC timestamps.
- [ ] `q` searches only non-secret textual fields such as path, user, user-agent, and request ID.
- [ ] Export supports at least JSONL and CSV.
- [ ] Export can return ZIP for large datasets.
- [ ] ZIP export contains exactly one data file in the selected format and MAY include an additional metadata manifest.
- [ ] Export endpoint supports the same filters (`from`, `to`, `user`, `ip`, `q`, `status`, `method`) as listing.
- [ ] No endpoint exposes audit data from other workspaces.

## 7. UI Behavior

### 7.1 Audit Logs View
- [ ] Audit logs are shown in a dedicated "Audit Logs" view (not as file tree folder).
- [ ] View supports filtering by date range (`from`/`to`), user, IP, search string, status, and method.
- [ ] View supports pagination.
- [ ] View supports export action.
- [ ] Export uses the currently selected filters (`from`, `to`, `user`, `ip`, `q`, `status`, `method`).

### 7.2 Configuration Modal
- [ ] Field: retention duration in days.
- [ ] Field: logging mode selector (`all`, `writes-only`, `auth-only`, `deployment-only`, `errors-only`).
- [ ] Field: exclude-health-endpoints toggle.
- [ ] Actions: Save and Cancel.

### Acceptance Criteria
- [ ] Changes are saved via API.
- [ ] Invalid configurations are rejected with clear errors.

## 8. Security & Privacy
- [ ] No plaintext secrets are ever written.
- [ ] Masking is identical to SSE masking behavior.
- [ ] Audit records are workspace-scoped.
- [ ] No cross-workspace visibility is possible.
- [ ] Access to audit read/export/config endpoints is RBAC-protected.
  - [ ] Anonymous sessions can access only the audit data of the workspace bound to their current session.
  - [ ] Authenticated users can access only workspaces they own, unless a future higher-order role is introduced explicitly.

### Acceptance Criteria
- [ ] Secret leakage through audit logging is prevented.
- [ ] Workspace isolation is enforced for read and export.

## 9. Performance Requirements
- [ ] Audit writing is non-blocking with respect to the request path; long-lived responses such as SSE MUST NOT emit per-chunk audit writes.
- [ ] DB writes do not materially delay API responses.
- [ ] Cleanup/export do not block active requests.
- [ ] Background jobs handle heavy cleanup/export work.
